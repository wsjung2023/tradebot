결론부터 말하면, 지금 증상은 “진짜 IP 차단 문제”라고 단정하면 안 됩니다.
IP 화이트리스트 정책은 키움 공식 문서에 실제로 있고, 허용된 IP에서만 API 요청을 받는다고 나와 있습니다. 다만 당신이 올린 소스는 키움 공식 REST 스펙과 핵심 부분이 여러 군데 어긋나 있어서, 이 상태에선 IP가 멀쩡해도 실패할 수 있는 구조입니다. 즉, IP 차단 가능성은 보조 원인이고, 1순위 원인은 API 사양 불일치로 보는 게 맞습니다.

소스 분석 결과, 가장 치명적인 부분은 이겁니다.

도메인이 틀렸습니다.
키움 공식 문서는 운영 도메인을 https://api.kiwoom.com, 모의투자 도메인을 https://mockapi.kiwoom.com로 안내합니다. 그런데 현재 서버 코드는 https://openapi.kiwoom.com:9443에 고정돼 있습니다.
server/services/kiwoom/kiwoom.base.ts:116
server/routes/account.routes.ts:157~170
특히 스크린샷상 선택 계좌가 모의투자인데, 현재 코드는 모의투자여도 mockapi로 안 가고 그냥 같은 baseURL을 씁니다. 이건 꽤 치명적입니다. 공식 문서상 모의투자는 별도 도메인입니다.

토큰 발급 스펙이 틀렸습니다.
키움 공식 문서는 토큰 발급 URL을 /oauth2/token, 요청 필드를 appkey, secretkey, 응답 필드를 token으로 안내합니다. 그런데 현재 코드는 /oauth2/tokenP에 appsecret을 보내고, 응답도 access_token을 기대합니다.
server/services/kiwoom/kiwoom.base.ts:140~151
client/src/lib/kiwoom-client.ts:38~64
이건 솔직히, 문 앞에서 초인종 대신 전자레인지 버튼 누르는 수준입니다. 집이 안 열리는 게 당연한 쪽에 가깝습니다.

잔고조회 API 형식이 키움이 아니라 한국투자증권(KIS) 스타일입니다.
현재 코드의 잔고조회 경로는 /uapi/domestic-stock/v1/trading/inquire-balance이고, TTTC8434R / VTTC8434R 같은 ID를 씁니다. 이 /uapi/... 계열은 한국투자증권 Open API에서 쓰는 경로 체계입니다. 반면 키움 공식 REST 문서는 국내주식 API를 /api/dostk/... 형식과 api-id 헤더 중심으로 안내합니다.
server/services/kiwoom/kiwoom.account.ts:43~63
client/src/lib/kiwoom-client.ts:113~133
즉 지금 코드는 이름은 키움인데, 속은 KIS 문법이 섞여 있는 상태입니다. 그래서 연결 실패가 나는 게 오히려 자연스럽습니다.

CORS 경고는 현재 zip 기준으로는 진단이 틀렸을 가능성이 큽니다.
현재 대시보드 훅은 브라우저가 키움으로 직접 붙는 게 아니라, 먼저 당신 서버의 /api/accounts/:accountId/fetch-balance를 호출합니다.
client/src/hooks/use-kiwoom-balance.ts:27~45
그래서 현재 코드 기준 대시보드 잔고조회 실패를 “브라우저 CORS 때문”이라고 보는 건 맞지 않습니다. 게다가 스크린샷의 오류 문구는 현재 zip 소스에서는 검색되지 않았습니다. 이건 높은 확률로

배포된 앱이 zip과 다른 구버전이거나

서비스워커/캐시가 오래된 번들을 보여주고 있거나

이전 직접호출 로직이 살아 있는 다른 배포본
중 하나입니다.

그래서 지금 판단은 이렇게 정리하는 게 정확합니다.

확정적 문제: 키움 공식 REST 스펙과 현재 코드가 불일치

높은 가능성: 배포본과 현재 zip 소스가 다름

가능성 있음: 서버 IP 미등록 또는 서버 outbound 제한

아직 증명 안 됨: “IP 차단이 진짜 원인이다”

그리고 중요한 포인트 하나.
현재 코드의 /api/kiwoom/test-connection도 정확한 증명 도구가 아닙니다. 이 라우트는 openapi.kiwoom.com:9443만 찍어보는데, 키움 공식 문서가 안내하는 REST 운영/모의 도메인은 api.kiwoom.com, mockapi.kiwoom.com입니다. 즉 이 테스트는 지금 기준으로 잘못된 대상을 향한 건강검진에 가깝습니다.

정확한 원인 판별 우선순위는 이렇게 가야 합니다.

첫째, 키움 REST 스펙으로 코드부터 바로잡아야 합니다.
도메인, 토큰 URL, 요청 필드명, 응답 필드명, 계좌조회 path, 헤더명을 전부 키움 문서 기준으로 맞춰야 합니다.

둘째, 모의투자는 무조건 mockapi.kiwoom.com으로 분기해야 합니다.
지금처럼 accountType에 따라 TR 비슷한 값만 바꾸는 방식은 안 됩니다.

셋째, 그 다음에야 IP 문제를 검증해야 합니다.
실계좌라면 키움 포털에 등록된 허용 IP와 현재 Replit 서버의 공인 IP가 일치하는지 확인해야 합니다.
반대로 모의투자에서 mockapi 토큰 발급이 정상 통과하면, 적어도 “네트워크 자체가 막혀서 아무것도 안 된다”는 가설은 크게 약해집니다.

넷째, 배포본 동기화를 봐야 합니다.
스크린샷 문구가 현재 zip에 없으니, 지금 눈앞의 앱은 소스와 다른 세계선에 서 있을 가능성이 큽니다.

한 줄로 박아두면 이렇습니다.

지금은 IP 차단을 의심할 단계가 아니라, 먼저 “키움 API로 보이는 KIS 코드”를 걷어내야 하는 단계입니다.
이걸 안 고치고 IP만 붙잡으면, 자물쇠는 다른 회사 건데 열쇠에 윤활유만 뿌리는 꼴이 됩니다.

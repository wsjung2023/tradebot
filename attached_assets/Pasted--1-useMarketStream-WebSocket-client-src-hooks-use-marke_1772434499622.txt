본범 1

클라이언트 useMarketStream() 이 WebSocket을 스스로 계속 갈아엎는 구조다.

파일:

client/src/hooks/use-market-stream.ts

핵심 라인:

connect() 안에서 setConnectionStatus('connecting') 함

ws.onopen 에서 setConnectionStatus('online') 함

그런데 connect의 의존성에 connectionStatus, retryCount 가 들어가 있음

그리고 아래 effect가 [connect, cleanup, symbols.length] 를 의존성으로 가지고 return cleanup 함

즉 구조가 이렇게 돼:

effect가 connect() 실행

connect() 가 connectionStatus 바꿈

connect 함수 identity가 바뀜

effect 재실행

이전 effect의 cleanup() 이 방금 만든 소켓을 닫음

다시 connect()

또 열고 또 닫고 또 재연결

이건 그냥 우아한 reconnect가 아니라,
소켓 생성/종료/타이머 생성/종료가 반복되는 churn 구조야.

문제 지점:

client/src/hooks/use-market-stream.ts:65-80

client/src/hooks/use-market-stream.ts:89-116

client/src/hooks/use-market-stream.ts:163-189

client/src/hooks/use-market-stream.ts:235-241

특히 client/src/hooks/use-market-stream.ts:197 의 dependency 배열

이건 진짜 냄새가 강하다.
한마디로:

연결 상태를 바꾸는 함수가, 그 상태 변화 때문에 자기 자신을 다시 만들고, 그 때문에 effect cleanup이 소켓을 다시 죽이는 구조

이건 메모리 누수 + 소켓 폭증 + 재연결 폭주 후보 1순위다.

본범 2

서버에서 실시간 시세 업데이트를 setInterval(async ...) 로 1초마다 때리는데, 이전 작업이 안 끝나도 다음 작업을 또 시작한다.

파일:

server/market-data-hub.ts

핵심 라인:

startPriceUpdates()

setInterval(async () => { ... }, 1000)

문제 지점:

server/market-data-hub.ts:191-230

여기서 매 1초마다:

getStockPrice(symbol)

getStockOrderbook(symbol)

를 await 하면서 돌고 있음.

그런데 키움 axios timeout이:

server/services/kiwoom.service.ts:173-176

server/services/kiwoom/kiwoom.base.ts:115-118

여기서 10초야.

즉 최악의 경우,

1초마다 새 작업 시작

각 작업은 최대 10초 매달림

이전 tick 안 끝났는데 다음 tick 계속 시작

pending promise / HTTP 요청 / 응답 버퍼 / JSON 객체 / ws send payload 가 계속 쌓임

이건 전형적인 비동기 interval 중첩 폭증이다.

쉽게 말하면:

주방장이 10초 걸리는 요리를 하는데
벨은 1초마다 울리고
새 주문을 계속 받는 구조

그러니 1분쯤 지나면 주방이 불난다.
네가 말한 “65초쯤 죽는다” 랑 아주 잘 맞는다.
이건 65초 하드코딩이 아니라, 누적 붕괴 시점이 대략 그쯤이라는 뜻.

공범 3

WebSocket upgrade를 두 번 건드리고 있다.

파일:

server/routes/index.ts

문제 지점:

new WebSocketServer({ server: httpServer, path: "/ws/market" })

동시에 또 httpServer.on("upgrade", ...) 안에서 wss.handleUpgrade(...)

라인:

server/routes/index.ts:33-35

server/routes/index.ts:40-52

이 구조는 꽤 위험하다.
보통 ws에서:

server + path 로 붙이거나

noServer: true + 수동 handleUpgrade 하거나

둘 중 하나를 써야 한다.

지금처럼 둘 다 걸면:

업그레이드 처리 중복

소켓 lifecycle 꼬임

listener/sock resource 누수

내부 ws 에러

같은 문제가 생길 수 있다.

이건 단독 본범이라기보다,
위 1번/2번이 이미 불붙인 상태에서 휘발유 붓는 역할이다.

왜 65초쯤 터지냐

정확히 말하면 코드에 65초 타이머는 없다.
대신 조합이 이렇다:

클라이언트가 WebSocket을 안정적으로 1개 유지 못하고 흔듦

서버는 구독 종목에 대해 1초마다 새 async polling 시작

각 polling은 최대 10초까지 살아 있음

reconnect / duplicate connection / duplicate subscription 가능성이 겹침

pending 요청 + ws 객체 + 타이머 + payload 객체가 누적

Replit 메모리 한계 근처에서 앱이 “뒤짐”

즉,
65초는 원인이 아니라 붕괴 타이밍이야.

내가 보는 우선순위
1순위 본범

client/src/hooks/use-market-stream.ts

이건 거의 확정급이다.
특히 dependency 설계가 너무 위험해.

핵심 문제

connect가 상태를 바꾸는데

connect가 그 상태에 의존하고

effect가 connect에 의존하며 cleanup에서 소켓을 닫음

이건 구조적으로 불안정하다.

2순위 본범

server/market-data-hub.ts

핵심 문제

setInterval(async ..., 1000)

이전 루프 완료 여부 확인 없음

in-flight 작업 개수 제한 없음

backpressure 없음

구독 수 증가 시 선형이 아니라 사실상 누적 폭증

3순위 공범

server/routes/index.ts

핵심 문제

WebSocketServer({ server, path }) 와

httpServer.on("upgrade") + handleUpgrade 를 동시에 사용

딱 잘라서 말하면

네 앱의 **메모리 누수/폭증 원인의 중심은 “실시간 시세 WebSocket + 서버 폴링 구조”**다.
그리고 그중에서도:

클라이언트는 reconnect churn

서버는 overlapping async polling

이 두 개가 거의 메인이다.

지금 바로 확인하면 확정되는 체크포인트

코드 수정 전에 로그만 박아도 바로 드러난다.

찍으면 계단식으로 오를 가능성이 높다.

가장 강한 수정 방향

원인 분석만 말하면 답답하니까, 바로 핵심만 적자면:

클라이언트

useMarketStream에서

connect dependency에서 connectionStatus, retryCount 제거

effect가 connect 바뀔 때마다 cleanup로 소켓 닫지 않게 구조 변경

소켓 인스턴스는 한 번 열고 필요할 때만 재연결

reconnect는 ref 기반으로 관리

서버

market-data-hub.ts에서

setInterval(async ...) 금지

이전 업데이트가 끝나지 않았으면 다음 tick skip

isUpdating 플래그 또는 self-scheduling setTimeout

symbol별 / 전체 in-flight 제한

가능하면 price/orderbook 병렬화도 제한적으로 관리

업그레이드 처리

ws 연결 방식 한 가지로 통일
(noServer: true 수동 처리 or server/path 자동 처리 중 하나만)

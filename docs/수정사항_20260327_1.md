## 2. 에러 출처 추적 (핵심 발견)

### 에러 메시지 형식 비교

에러 메시지 형식이 두 개 존재함:

```
# A. 에이전트 에러 형식 (kiwoom-agent.py 175번 줄)
"키움 WS 오류: 로그인 인증이 들어오기 전에 다른 접속이 들어왔습니다. (code: 100013)"

# B. Legacy 서버 에러 형식 (kiwoom.condition.ts 68번 줄)
"조건검색 오류 100013: 로그인 인증이 들어오기 전에 다른 접속이 들어왔습니다."
```

브라우저에 표시된 에러:

```
"조건검색 오류 100013: 로그인 인증이 들어오기 전에..."
```

**결론: 브라우저 에러는 형식 B와 완전 일치 → Legacy 서버 코드에서 발생하는 에러임**

에이전트가 아니라 **Replit 서버가 직접 키움 WebSocket에 연결하다 발생하는 에러**다.

## 4. 원인 확정

### 직접 원인 (Primary)

`server/services/user-kiwoom.service.ts` 158~163번 줄:

```typescript
// 현재 코드 (문제)
async getConditionList(userId: string) {
  try {
    const response = await callViaAgent(userId, "condition.list", {});
    return response?.output ?? response ?? [];
  } catch (error) {
    if (error instanceof AgentTimeoutError) throw error;
    const legacyKiwoom = await this.getLegacyServiceForUser(userId);   // ← 문제
    const response = await legacyKiwoom.getConditionList();            // ← 문제
    return response?.output ?? response ?? [];
  }
}
```

에이전트에서 에러 발생 시, **Replit 서버 자체가 Legacy WebSocket으로 키움에 접속 시도**함.
이것이 에러 100013의 직접적인 원인임. Legacy fallback이 실행되어선 안 됨.

### 간접 원인 (Secondary)

`agent/kiwoom-agent.py` 189~192번 줄:

```python
# 현재 코드 (문제)
header={
    "api-id": api_id,
    "Authorization": f"Bearer {token}",
},
```

에이전트가 Python `websocket-client` 라이브러리에 딕셔너리 형식으로 헤더를 전달함.
이 방식은 HTTP Upgrade 요청에 헤더가 포함되지 않을 수 있어 키움 인증 실패의 원인이 됨.
결과적으로 에이전트가 에러를 반환하고, 이것이 Legacy fallback을 촉발시킴.

### 왜 Legacy도 100013을 받는가

키움 WebSocket은 **계정(토큰) 당 하나의 WebSocket 연결만 허용**함.
에이전트가 먼저 WebSocket을 열었다가 에러를 받고 닫는 과정에서,
그 직후 Replit 서버의 Legacy 코드가 또 WebSocket을 열면:
- 에이전트 연결이 아직 완전히 닫히지 않은 경우 → 동시 접속 충돌
- 또는 Replit의 유동 IP가 키움 WebSocket 허용 IP 목록에 없는 경우

둘 중 하나이며, 결과는 동일하게 에러 100013.

## 5. 수정 지시 — 파일 2개 수정

> **수정 우선순위**: 수정 1이 더 중요하고 즉각적인 효과가 있음. 수정 2는 에이전트 자체 성공률을 높임.

---

### 수정 1 (필수, 즉시 효과) — `server/services/user-kiwoom.service.ts`

**위치**: 154~164번 줄 `getConditionList` 함수

**현재 코드 (154~164번 줄)**:

```typescript
  async getConditionList(userId: string) {
    try {
      const response = await callViaAgent(userId, "condition.list", {});
      return response?.output ?? response ?? [];
    } catch (error) {
      if (error instanceof AgentTimeoutError) throw error;
      const legacyKiwoom = await this.getLegacyServiceForUser(userId);
      const response = await legacyKiwoom.getConditionList();
      return response?.output ?? response ?? [];
    }
  }
```

**수정 후 코드**:

```typescript
  async getConditionList(userId: string) {
    const response = await callViaAgent(userId, "condition.list", {});
    return response?.output ?? response ?? [];
  }
```

**변경 내용**:
- try-catch 블록 전체 제거
- Legacy fallback (`legacyKiwoom.getConditionList()`) 제거
- 에이전트 에러는 그대로 상위로 전파되어 클라이언트에 정확한 에러 메시지 표시됨
- `callViaAgent` 자체가 내부에서 타임아웃 처리를 함

---

### 수정 2 (권장, 에이전트 자체 성공률 향상) — `agent/kiwoom-agent.py`

**위치**: 187~196번 줄 `kiwoom_ws_request` 함수 내 `WebSocketApp` 생성 부분

**현재 코드 (189~192번 줄)**:

```python
    ws_app = websocket.WebSocketApp(
        ws_url,
        header={
            "api-id": api_id,
            "Authorization": f"Bearer {token}",
        },
        on_open=on_open,
        on_message=on_message,
        on_error=on_error,
    )
```

**수정 후 코드**:

```python
    ws_app = websocket.WebSocketApp(
        ws_url,
        header=[
            f"api-id: {api_id}",
            f"Authorization: Bearer {token}",
        ],
        on_open=on_open,
        on_message=on_message,
        on_error=on_error,
    )
```

**변경 내용 요약**:

| 항목 | 변경 전 | 변경 후 |
|------|---------|---------|
| `header` 타입 | `dict` (`{}`) | `list` (`[]`) |
| `api-id` 형식 | `"api-id": api_id` | `f"api-id: {api_id}"` |
| `Authorization` 형식 | `"Authorization": f"Bearer {token}"` | `f"Authorization: Bearer {token}"` |

## 6. 수정 효과 예측

### 수정 1만 적용한 경우

```
[에이전트 WebSocket 성공] → 조건식 목록 정상 반환  ✓
[에이전트 WebSocket 실패] → Legacy fallback 없음 → 에이전트 에러 메시지 클라이언트 전달
                            (에러 형식: "키움 WS 오류: ...")
```

Legacy fallback 자체가 제거되므로 100013 에러는 즉시 사라짐.
단, 에이전트 WebSocket 자체가 실패하면 다른 에러 메시지가 표시됨.

### 수정 1 + 수정 2 모두 적용한 경우

```
[에이전트 WebSocket 성공 (헤더 리스트로 인증)] → 조건식 목록 정상 반환  ✓
[에이전트 WebSocket 실패] → 정확한 에러 메시지 클라이언트 전달
```

에이전트 WebSocket 성공률도 높아지므로 완전한 해결 가능성이 가장 높음.

## 7. 수정 후 검증 방법

1. 에이전트를 재시작
2. 서버를 재시작
3. 화면에서 **조건검색** 메뉴 이동
4. **"HTS 조건식 불러오기"** 버튼 클릭

**성공 조건**:
- 에러 없이 키움 HTS에 등록된 조건식 목록이 드롭다운에 표시됨
- 에이전트 콘솔에 `condition.list 조회 완료` 로그 출력

**수정 1만 하고 실패한 경우**:
- 에러 메시지가 `"조건검색 오류 100013:"` 이 아니라 `"키움 WS 오류: ..."` 형식으로 바뀌어야 함
- 만약 여전히 `"조건검색 오류 100013:"` 형식이라면 수정 1이 제대로 반영되지 않은 것
- `"키움 WS 오류: ..."` 형식으로 바뀌었다면 에이전트 WebSocket 자체 문제 → 수정 2 적용

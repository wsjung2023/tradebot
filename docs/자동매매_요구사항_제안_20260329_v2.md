# 자동매매 요구사항/아키텍처 제안 v2 (2026-03-29)

## 0. 결론(요약)
- 사용자 요구는 타당하며, 현재 구조 대비 **운영요건(항상 켜짐, 인증만으로 제어, 끊김 알림, 실시간 모니터링, 규칙가중치 커스터마이징)**이 더 높은 수준이다.
- 따라서 단일 루프 기반에서 **Job 분리 + 상태 영속화 + 이벤트 기반 알림** 구조로 전환하는 것이 맞다.

## 1. 사용자가 제시한 프로세스 플로우에 대한 판단

### 요구 1) 프론트 로그인 후 내 계정 Job 가동
- 맞다. 단, "로그인"은 제어 권한 확인 용도이고, 실제 Job 실행주체는 백엔드 워커여야 한다.

### 요구 2) 한 번 가동되면 끄기 전까지 계속 + 토큰 만료/오류 시 알림
- 맞다. Job 상태를 DB에 영속화하고, 워커는 재시작 시 자동 복구해야 한다.
- 인증 토큰/키움 세션 만료/주문 실패는 `job_events`에 기록하고, 사용자에게 알림(WebSocket + 푸시 + 이메일 선택) 발송.

### 요구 3) 프론트 로그아웃/브라우저 종료와 무관하게 백엔드 지속
- 100% 맞다. 세션과 잡 생명주기는 분리되어야 한다.

### 요구 4) 재로그인 시 성과/실시간 타겟/승률 모니터링
- 맞다. 대시보드에서 기간 필터(당일/1주/1달/6달/1년) + 실시간 상태 스트림 제공이 필요.

## 2. 권장 Job 분리(핵심)

### Job A. Universe Scanner (종목검색기)
- 입력: 뒷차기2 조건검색기 결과
- 출력: `candidate_queue`
- 주기: 10~30초 또는 1분

### Job B. Signal Engine (평가/점수화)
- 입력: `candidate_queue` + 보유종목
- 규칙: 레인보우/섹터모멘텀/뉴스재료/재무개선/경보기호/유동성/CL폭/이평결합
- 출력: `signal_events` (buy/sell/hold, confidence, reason)

### Job C. Execution Engine (주문/리스크)
- 입력: `signal_events`
- 기능: 중복주문방지, 주문한도, 분할매수/분할매도, 실패재시도
- 출력: `orders`, `fills`, `positions`

### Job D. Learning Engine (장마감)
- 입력: 거래성과 + 피처
- 기능: 자동 가중치 튜닝(옵션), 수동고정 모드와 공존

## 3. 타겟 종목 조건(사용자 제안) 반영 방식

사용자 제안 1~8 항목은 아래처럼 "점수카드"로 구현하는 것이 실무적으로 안정적이다.

- Hard Filter (필수통과)
  - 거래대금/유동성 하한
  - 시장경보/관리종목 등 제외 규칙
- Soft Score (가중합)
  - 레인보우 위치/CL폭
  - 중심섹터 적합도(뉴스/이슈)
  - 재료 생존성(최근 뉴스/공시 감쇠)
  - 2~3년 재무개선
  - 이평/추세 결합지점

최종점수 = Σ(항목점수 × 가중치)
- **수동모드**: 사용자가 가중치 고정
- **자동모드**: 학습엔진이 가중치 업데이트(최소/최대 범위 제한)

## 4. 분할매수/분할매도 설정 모델

### 매수
- CL 라인별 고정금액 또는 비율
- 예: CL1~CL5 각 20% 또는 금액 N원 동일

### 매도
- 목표수익률 구간별 분할
- 예: +3%(20%), +5%(30%), +8%(50%)
- 전역 설정 + 종목별 override 지원

## 5. 운영 신뢰성(필수)

- Job Heartbeat: 각 Job이 N초마다 heartbeat 기록
- Watchdog: heartbeat 누락 시 자동 재기동
- Event Log: `started/stopped/error/retried/token_expired`
- 알림: 심각도별 사용자 통지
- 재시작 복구: 서버 재기동 후 `job_state=running`인 잡 자동 복원

## 6. 단계별 구현 우선순위

### Phase 1 (필수)
1) Job 상태 영속화 + watchdog + 알림
2) 프론트 대시보드 기간별 성과/승률/실시간 상태
3) Signal/Execution 분리(최소)

### Phase 2 (고도화)
4) Scanner 분리 + 큐 기반 처리
5) 수동/자동 가중치 동시 지원 UI
6) 분할매수/분할매도 템플릿(전역/종목별)

### Phase 3 (최적화)
7) 학습엔진 온라인 튜닝, 드리프트 감시
8) 백테스트/리플레이 검증 파이프라인

## 7. 내 의견(직접 답변)
- 사용자님 생각이 맞다. 특히 **"로그아웃해도 백엔드에서 계속 돌아야 함"**, **"토큰 만료/오류 알림"**, **"스캐너와 실행 분리"**는 운영에서 반드시 필요한 축이다.
- 다만 한번에 전부 바꾸면 리스크가 크므로, Phase 1부터 순차 적용을 권장한다.

# 자동매매/배치잡 점검 리포트 (2026-03-29)

## 1) 현재 구현 상태 요약

### A. 장중 감시 시간 (KST 09:00~15:30)
- `isKoreanMarketOpen()`에서 KST 오프셋(+9h)을 적용한 뒤,
  - 주말(토/일) 제외
  - `09:00 <= 현재시각 <= 15:30`일 때만 `true`
- 자동매매 사이클(`executeTradingCycle`)은 매 실행 시 이 함수를 먼저 검사하고, 장외면 스킵한다.

### B. 감시 주기
- 자동매매 워커의 기본 스케줄은 `* * * * *` (1분 주기).
- 따라서 "틱 단위"가 아니라 "분 단위 폴링"이다.

### C. 장중 종목 탐색/매수/매도
- 활성 AI 모델 목록 조회 -> 모델별 활성 조건식 조회 -> 조건검색 실행 -> 조건 결과 종목(최대 10개) 평가 순서로 동작.
- 종목 평가는 AI 신뢰도/재무/유동성 + 레인보우 시그널을 합쳐서 `buy/sell/hold` 판단.
- 매수 시 주문 생성 후 키움 주문 전송, 매도 시 보유수량 확인 후 주문 전송 및 성과 업데이트 수행.

### D. 배치잡 On/Off
- 서버 API로 잡 조회/시작/중지/주기변경/즉시실행이 제공된다.
- 관리자 페이지에서 상태(`running/stopped`)를 기준으로 시작/중지 버튼이 노출된다.

## 2) 잡 구성에 대한 의견

질문에서 제시한 3개 축은 타당하다.

1. 시그널 포착 + 거래 실행
2. 종목검색/타겟팅
3. 장마감 후 학습

현재 코드는 (1)과 (2)가 사실상 **하나의 트레이딩 사이클 안에 결합**되어 있다.
- 장점: 단순하고 구현 비용이 낮다.
- 단점: 스캔/의사결정/주문이 강결합되어 장애 분리가 어렵고, 고빈도 확장에 불리하다.

운영 고도화 관점에서는 아래 분리가 더 바람직하다.
- Job A: 유니버스 스캐너(조건검색 전용)
- Job B: 시그널 엔진(보유/관심종목 공통 의사결정)
- Job C: 주문 실행기(리스크 한도/중복주문 방지 포함)
- Job D: 장마감 학습

다만 지금 단계에서 반드시 즉시 분리해야 하는 것은 아니며,
현재 트래픽/전략 복잡도/장애 빈도를 보고 단계적으로 분리하는 것이 현실적이다.

## 3) 1분 폴링이 문제인가?

전략 의존적이다.

- 스윙/단기추세(분~시간 단위) 전략: 1분 폴링도 실무에서 충분히 사용 가능.
- 초단타/호가 기반 전략: 1분은 늦다(슬리피지/체결 불리).

따라서 "문제"라기보다 "전략 적합성"의 문제다.

권장 기준:
- 목표 보유시간이 길고, 진입 신호가 분봉/일봉 기반이면 1분 폴링 유지 가능.
- 체결 타이밍 민감도가 높으면 tick/websocket 기반 신호수집으로 전환 고려.

## 4) 결론

- 현재 구현은 "장중(09:00~15:30) 분주기 감시 -> 조건검색 -> 매수/매도"라는 큰 틀은 맞다.
- 배치잡 On/Off 제어도 서버/API/UI 레벨에서 동작 구조가 있다.
- 다만 아키텍처적으로는 스캔/시그널/주문의 분리 여지가 크고,
  전략이 고빈도로 갈수록 1분 폴링은 한계가 명확하다.

## 5) 추가 답변: "종목검색기 타겟팅 Job"은 따로 있나?

결론부터 말하면 **지금은 독립 Job으로 분리되어 있지 않고, 트레이딩 사이클 내부 단계로 포함**되어 있다.

- `executeTradingCycle()` -> `processModel()` -> `processCondition()` 흐름에서
  조건검색(`runCondition`)으로 종목을 찾고, 그 결과를 바로 평가/주문으로 넘긴다.
- 즉, "종목검색 전용 배치잡"이 따로 스케줄링되어 동작하는 구조는 아니다.

### 왜 이게 중요하나
- 현재 구조는 단순해서 빠르게 운영 가능하지만,
  스캔 병목/주문 병목/AI 병목을 따로 모니터링하거나 재시도 정책을 분리하기 어렵다.

### 권장안(단계적)
1. **스캐너 Job 분리**: 조건검색만 수행해 후보 종목 큐/테이블에 적재  
2. **시그널 Job 분리**: 후보 + 보유종목에 대해 매수/매도 시그널만 계산  
3. **실행 Job 분리**: 주문/체결/리스크 제한 담당

이렇게 하면 질문하신 "종목검색기로 타겟팅하는 Job"이 명확히 독립되며,
장중 부하가 커져도 각 단계를 개별 확장/장애격리할 수 있다.

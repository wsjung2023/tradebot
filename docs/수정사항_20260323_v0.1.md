## 0. Codex 작업 지시의 기본 원칙

아래 원칙을 **반드시 지켜서 수정**한다.

### 원칙 A. Kiwoom 접근 경로를 한 방향으로 통일할 것
현재 프로젝트는 Kiwoom 연동이 다음 네 갈래로 분산되어 있다.

1. 집 PC 에이전트 경유 (`callViaAgent`)  
2. 서버 환경변수 기반 `getKiwoomService()` 싱글톤  
3. 사용자 DB 저장 키 기반 `new KiwoomService(...)`  
4. WebSocket 실시간 시세용 `MarketDataHub(getKiwoomService())`

이 상태에서는 화면마다 서로 다른 자격증명/세션/데이터 원천을 사용하게 되어, 사용자 입장에서 “어떤 화면은 되고 어떤 화면은 안 되는” 분열이 발생한다.

**이번 수정의 1차 원칙은 `사용자 단위 Kiwoom 기능은 agent 경유`를 표준으로 삼는 것**이다.

- 종목 검색
- 종목 기본정보
- 차트
- 관심종목 시세 새로고침
- 조건식 목록/실행

위 기능은 가능하면 모두 `callViaAgent(userId, jobType, payload)` 계열로 수렴시킨다.

### 원칙 B. 실패를 로딩으로 위장하지 말 것
현재는 실패/빈데이터/로딩이 한 UI 상태로 섞여 있다.  
이 패턴은 디버깅을 망치고, 사용자는 “계속 로딩 중”으로 오해한다.

따라서 각 화면은 아래 상태를 분리한다.

- loading
- success(hasData)
- success(empty)
- error
- agent timeout

### 원칙 C. 종목 입력은 “문자열”이 아니라 “선택된 종목 객체” 중심으로 다룰 것
`stockCode`, `stockName`, `currentPrice`를 서로 독립 입력값으로 관리하면 항상 꼬인다.  
공통 선택 모델 예시:

```ts
interface SelectedStock {
  stockCode: string;
  stockName: string;
  currentPrice?: number;
  marketName?: string;
}
```

모든 화면은 이 객체를 기준으로 동기화한다.

### 원칙 D. 기존 구형 KiwoomService는 “대체 경로”가 아니라 “제거 대상”으로 본다
이번 수정에서 최소한 다음 라우트/기능은 구형 서비스 의존을 줄이거나 제거한다.

- `/api/kiwoom/conditions`
- `/api/kiwoom/conditions/:seq/run`
- WebSocket 실시간 시세 허브

### 원칙 E. 무관한 리팩토링 금지
Codex는 아래 범위 밖 수정은 하지 말 것.

- UI 디자인 전면개편 금지
- DB 스키마 무관 변경 금지
- 이름만 바꾸는 대규모 리네이밍 금지
- 동작 원인과 무관한 정리성 리팩토링 금지

**지금 목표는 “연결 복원”이지 “예쁘게 재작성”이 아니다.**

## 1. 테스트 결과(PPTX) 기준 핵심 증상

업로드된 PPTX의 테스트 문구 요약:

1. 종목명 Like 검색이 안 됨  
2. 차트가 “데이터 로딩 중” 상태에서 안 나옴  
3. 종목코드/종목명 검색이 전혀 안 됨  
4. 종목코드↔종목명↔현재가가 자동 연결되지 않는데도 “모든 필드를 입력” 요구  
5. 내 키움 조건검색기를 불러오지 못함

이 다섯 증상은 각각 따로 보이지만, 실제로는 아래 두 축으로 이어진다.

- **축 1: 종목 선택 파이프라인이 느슨하고 일관성이 없다**
- **축 2: Kiwoom 호출 경로가 agent / server-env / user-key / old-service 로 분열되어 있다**

### 구조 진단 1. 관심종목 새로고침 API 경로 불일치

### 구조 진단 2. 종목 검색 구현이 퇴화함
- 현재 에이전트 검색: `agent/kiwoom-agent.py:242-259`
- 과거 더 나은 구현: `server/services/kiwoom_OLD/kiwoom.market.ts:236-274`

현재 에이전트 검색은 `ka10002`와 `{"stk_nm": keyword}`만 사용한다. 이 구현은:

- 6자리 종목코드 exact 검색을 별도로 처리하지 않음
- 종목명 부분검색(Like)을 보장하지 않음
- 캐시 기반 부분검색 보완이 없음

반면 구형 구현에는:

- 코드 exact 검색
- 이름/코드 includes 검색

이 있었다.

즉 신형 agent 경로로 바뀌면서 검색 품질이 떨어졌다.

### 구조 진단 3. 차트 UI가 실패를 로딩으로 숨김

### 구조 진단 4. agent 차트 파싱과 old Kiwoom 차트 파싱이 서로 다름
- 현재 에이전트 차트: `agent/kiwoom-agent.py:199-226`
- 구형 차트: `server/services/kiwoom_OLD/kiwoom.market.ts:115-155`

차트 API 요청 파라미터와 응답 필드 해석이 서로 다르다.

- `upd_stkpc_tp: "1"`

이 차이 때문에 실제 응답이 와도 0 또는 빈 데이터로 변환될 가능성이 높다.

### 구조 진단 5. 조건검색은 “내 키움”이 아니라 서버 싱글톤 KiwoomService를 보고 있음
- `server/routes/formula.routes.ts:13, 16-33`
- `server/services/kiwoom_OLD/index.ts:65-77`

조건검색 라우트는 `getKiwoomService()`를 사용하며, 이 함수는 `process.env.KIWOOM_APP_KEY`, `process.env.KIWOOM_APP_SECRET` 기반 싱글톤 인스턴스를 만든다.

즉 “내 HTS 조건검색기”가 아니라 **서버 전체 공용 키움 연결**을 보게 된다. 이 구조에서는 사용자 개인 조건검색식이 안 보이는 것이 오히려 자연스럽다.

### 구조 진단 6. WebSocket 실시간 시세도 서버 env 기반 서비스에 묶여 있음
- `server/routes/index.ts:19-21`

즉 이 또한 agent나 사용자 저장 키가 아니라, 서버 싱글톤 KiwoomService를 사용한다.

### 구조 진단 7. AI 분석 화면은 입력 모델이 분리되어 있어 UX가 비정상적임
- `client/src/pages/ai-analysis.tsx:14-40`

현재 구조는 사용자가:

- 종목코드

- 현재가

를 모두 수동 입력해야 한다.  
하지만 앱 내부엔 이 셋을 자동으로 연결해주는 공통 종목선택 파이프라인이 없다.

1. keyword trim

## 8. 최종 판단

이 프로젝트의 현재 핵심 문제는 “버그가 많다”가 아니라, **Kiwoom 데이터 흐름이 통일되어 있지 않아 화면마다 서로 다른 세계를 본다**는 점이다.

따라서 이번 수정의 성공 기준은 단순히 버튼 하나 고치는 것이 아니라 아래 세 줄을 만족하는 것이다.

1. **검색 → 선택 → 차트/분석/관심종목 추가** 흐름이 하나로 이어질 것  
2. **내 Kiwoom 기준 기능**은 가능한 한 agent 경유로 통일될 것  
3. 실패는 실패로 드러나고, 로딩으로 위장되지 않을 것

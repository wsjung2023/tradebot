## 0. 전제: 집 PC 에이전트 의존 구조 이해

이 프로젝트의 **모든 키움 API 연동 기능은 집 PC 에이전트가 켜져 있어야 동작한다.**

```
[브라우저] → [Replit 서버] → [집 PC kiwoom-agent.py] → [키움증권 API]
```

- Replit 서버는 유동 IP이므로 키움 API에 직접 접근 불가
- 키움 API는 고정 IP 등록이 필수
- 따라서 조건검색, 차트, 종목 검색, 잔고 조회 전부 에이전트 경유
- 에이전트가 꺼져 있으면 `AgentTimeoutError` 발생 후 legacy fallback 시도하지만,
  legacy fallback도 동일하게 키움 API를 직접 호출하므로 고정 IP 없이는 실패

**"에이전트 없이 테스트"는 불가능하다. 이 전제 하에 아래 수정 작업을 진행한다.**

## 2. 이슈 #1 — 차트 수식이 차트에 그려지지 않음 (최우선)

### 현재 상태

`/api/chart-formulas/:id/evaluate`를 호출하면 서버는 아래 형태를 반환한다:

```json
{
  "stockCode": "005930",
  "period": "D",
  "formulaName": "나의MA20",
  "signalLine": {
    "color": "#8b5cf6",
    "name": "나의MA20",
    "values": [
      { "date": "20250101", "value": 12345 },
```

그런데 현재 `ChartFormulaList.tsx`는 이 결과를 **숫자와 색 점으로만 표시**한다:

## 3. 이슈 #2 — `legacyByUser` Map 메모리 누수

### 위치

파일: `server/services/user-kiwoom.service.ts`  
라인: 18번

```ts
private legacyByUser = new Map<string, CachedLegacyService>();
```

### 현재 문제

이 Map은 사용자가 처음 접속할 때 항목이 추가되고, **절대 지워지지 않는다.**  
사용자가 앱키를 변경하면 fingerprint가 달라지므로 새 항목이 추가되지만, 이전 항목은 그대로 남는다.  
서버가 장기 운영되면 사용자 수 × 키 변경 횟수만큼 인스턴스가 누적된다.

### 수정 목표

Map 크기를 일정 수 이하로 유지한다. 가장 오래된 항목부터 제거하는 LRU 방식을 적용한다.

### 수정 지시

#### (1) 마지막 접근 시간 추적 필드 추가

```ts
// 6번 줄 CachedLegacyService 타입 수정
type CachedLegacyService = {
  fingerprint: string;
  service: KiwoomService;
  lastUsed: number; // 추가
};
```

#### (2) 캐시 조회 시 lastUsed 갱신

```ts
// 36-38번 줄 수정
const cached = this.legacyByUser.get(userId);
if (cached?.fingerprint === fingerprint) {
  cached.lastUsed = Date.now(); // 추가
  return cached.service;
}
```

#### (3) 캐시 저장 시 lastUsed 포함

```ts
// 45번 줄 수정
this.legacyByUser.set(userId, { fingerprint, service, lastUsed: Date.now() });
```

#### (4) 최대 크기 제한 메서드 추가 (클래스 내부)

```ts
private readonly MAX_LEGACY_CACHE = 50;

private evictOldLegacyCache() {
  if (this.legacyByUser.size <= this.MAX_LEGACY_CACHE) return;

  const entries = Array.from(this.legacyByUser.entries())
    .sort((a, b) => a[1].lastUsed - b[1].lastUsed);

  const toRemove = entries.slice(0, entries.length - this.MAX_LEGACY_CACHE);
  for (const [key] of toRemove) {
    this.legacyByUser.delete(key);
  }
}
```

#### (5) 캐시 저장 직후에 호출

```ts
// 45번 줄 아래에 추가
this.legacyByUser.set(userId, { fingerprint, service, lastUsed: Date.now() });
this.evictOldLegacyCache(); // 추가
```

### 수용 기준

- 50명 이상 사용자가 접속해도 Map 크기가 50 이하로 유지됨
- 가장 최근에 사용한 사용자의 캐시는 보존됨

## 4. 이슈 #3 — WebSocket 시세 업데이트 순차 처리 병목

### 위치

파일: `server/market-data-hub.ts`  
라인: 187-191번

```ts
for (const [userId, symbols] of Array.from(demandByUser.entries())) {
  for (const [symbol, demand] of Array.from(symbols.entries())) {
    await this.pushSymbolSnapshot(userId, symbol, ...); // 순차 처리
  }
}
```

### 현재 문제

구독 종목이 많을 때 전부 순차적으로 `await`한다.  
예: 사용자 2명이 각 5개 종목 구독 중이면, 10번 API 호출을 하나씩 기다린다.  
UPDATE_INTERVAL이 2000ms인데 10번 호출이 그 안에 끝나지 않으면 다음 사이클이 지연된다.

### 수정 목표

사용자별로는 병렬로, 같은 사용자의 종목들도 병렬로 처리한다.

### 수정 지시

#### 현재 코드 (187-191번)

```ts
for (const [userId, symbols] of Array.from(demandByUser.entries())) {
  for (const [symbol, demand] of Array.from(symbols.entries())) {
    await this.pushSymbolSnapshot(userId, symbol, Array.from(demand.channels), Array.from(demand.clients));
  }
}
```

#### 수정 코드

```ts
// 사용자별로 병렬 → 각 사용자 내 종목별로도 병렬
await Promise.all(
  Array.from(demandByUser.entries()).map(([userId, symbols]) =>
    Promise.all(
      Array.from(symbols.entries()).map(([symbol, demand]) =>
        this.pushSymbolSnapshot(
          userId,
          symbol,
          Array.from(demand.channels),
          Array.from(demand.clients),
        )
      )
    )
  )
);
```

### 주의사항

- `pushSymbolSnapshot` 내부에 `try-catch`가 있으므로 한 종목 실패가 전체에 영향을 주지 않음
- 에이전트로 동시에 여러 요청이 가면 에이전트 측 부하가 높아질 수 있으므로,  
  구독 종목 수가 많을 경우 `p-limit` 같은 동시성 제한 라이브러리 사용 검토

### 수용 기준

- 5개 종목 구독 시 시세 업데이트 응답이 2초 이내에 완료됨
- 한 종목 API 호출 실패가 다른 종목 업데이트를 막지 않음

## 5. 이슈 #4 — `sync-financials`가 전역 `kiwoomService` 사용

### 위치

파일: `server/routes/formula.routes.ts`  
라인: 363-401번

다른 모든 API는 `userKiwoomService` (사용자 단위, agent 우선)를 사용하는데,  
`/api/stocks/sync-financials` 엔드포인트만 `getKiwoomService()` 전역 싱글톤을 사용한다.

- agent 경유가 아닌 서버 직접 키움 호출
- 일관성 없는 에러 처리 (다른 엔드포인트와 다른 실패 패턴)

### 수정 목표

`sync-financials`도 `userKiwoomService`를 통해 사용자 단위로 처리한다.  
재무 데이터 API를 `userKiwoomService`에 추가하거나, 임시로 agent-proxy를 직접 호출한다.

### 수정 지시

```python
def kiwoom_ws_request(api_id, payload, is_mock=None):
    """
    키움 WebSocket API 요청 (단발성 요청-응답)
    - 조건검색(ka10171, ka10172) 등에 사용
    - wss://api.kiwoom.com:10000/api/dostk/websocket
    """
    use_mock = KIWOOM_IS_MOCK if is_mock is None else is_mock
    token = get_kiwoom_token(is_mock=use_mock)
    if not token:
        raise ValueError("키움 WebSocket 토큰 없음")

    base_ws = "wss://mockapi.kiwoom.com:10000" if use_mock else "wss://api.kiwoom.com:10000"
    ws_url = f"{base_ws}/api/dostk/websocket"

    result = {"data": None, "error": None}

    def on_open(ws):
        ws.send(json.dumps(payload))

    def on_message(ws, raw):
        try:
            msg = json.loads(raw)
            # trnm == "REAL" 은 실시간 push이므로 무시
            if msg.get("trnm") == "REAL":
                return
            rc = msg.get("return_code")
            if rc is not None and rc != 0 and str(rc) != "0":
                result["error"] = f"키움 WS 오류: {msg.get('return_msg')} (code: {rc})"
            else:
                result["data"] = msg
        except Exception as e:
            result["error"] = str(e)
        finally:
            ws.close()

    def on_error(ws, error):
        result["error"] = str(error)
        ws.close()

    ws_app = websocket.WebSocketApp(
        ws_url,
        header={
            "api-id": api_id,
            "Authorization": f"Bearer {token}",
        },
        on_open=on_open,
        on_message=on_message,
        on_error=on_error,
    )

    ws_app.run_forever(ping_timeout=20)  # 20초 내 응답 없으면 종료

    if result["error"]:
        raise ValueError(result["error"])
    if result["data"] is None:
        raise TimeoutError(f"키움 WebSocket 응답 없음 (api_id={api_id})")

    return result["data"]
```

```python
def handle_condition_run(payload):

    payload: {"seq": "0"}  # 조건식 번호 (문자열 또는 정수)

      {

      }

      {
        "trnm": "CNSRREQ",
        "return_code": 0,

    try:
```

### 수용 기준

1. 에이전트 실행 후 로그에서 지원 jobType 목록에 `condition.list`, `condition.run` 표시됨
2. DB에 `condition.list` 작업 삽입 시 에러 없이 `done` 상태가 되고, `result` 컬럼에 조건식 목록 JSON 반환됨

4. 조건식 선택 후 "실행" 클릭 시 종목 리스트가 테이블에 표시됨
